# REKAP DISKUSI ARSITEKTUR & KONTEKS (BACA SEBELUM CODING)

Dokumen ini berisi rangkuman diskusi, keputusan arsitektur, dan tanya jawab (Q&A) yang sangat krusial antara Klien/User dengan System Architect sebelumnya. 
Agen Eksekutor WAJIB memahami sejarah ini agar tidak melakukan kesalahan fatal saat memanipulasi data FoxPro.

---

## 1. Topologi Jaringan & Keamanan Data
- **Server Original (192.168.1.100)**: Database MySQL 5.0 lawas (pastikan menggunakan `charset='utf8'` bukan `utf8mb4`). Ini adalah data murni/jujur operasional perusahaan. **TIDAK BOLEH ADA PROSES INSERT/UPDATE/DELETE KE SERVER INI.**
- **Server Pajak / Gemuk (192.168.1.200)**: Database tempat pelaporan pajak. Data dari Server Original ditarik, di-adjust/dimanipulasi oleh Python, lalu disimpan/ditimpa ke Server Gemuk ini. Auditor hanya akan memeriksa Server Gemuk.

## 2. Struktur Database Legacy (FoxPro)
- **Tabel Transaksi**: `DJUAL` (Detail Jual), `DRJUAL` (Retur Jual), `DBELI` (Detail Beli), `DRBELI` (Retur Beli), dan `BARANG` (Master Barang).
- **Identifier Struk (PENTING)**: Untuk mengelompokkan satu kesatuan struk/faktur, **JANGAN HANYA MENGGUNAKAN `TGL_JUAL`**. Gunakan kolom **`F_JUAL`** (Faktur Jual) untuk tabel penjualan, dan **`F_BELI`** untuk tabel pembelian.
- **Index Jual & Beli (`P_JUAL` / `P_BELI`)**: Di tabel global `SETOR_PAJAK`, nilai *Index* bukanlah persentase acak. Ia adalah persentase selisih/margin antara omset bersih dengan omset yang dilaporkan. Rumus aslinya: `((REAL_JUAL - R_JUAL) - JUAL) / (REAL_JUAL - R_JUAL) * 100`.

## 3. Q&A Krusial: Logika Penyesuaian Pajak (Adjustment PPN)

**Q: Mengapa kita tidak menghapus keseluruhan struk (Header) saat harus mengurangi omset?**
A: Sangat dilarang oleh klien! Menghapus struk utuh akan menyebabkan Loncatan Nomor Struk (Nomor Faktur bolong). Ini pantangan besar dalam audit pajak. Manipulasi/pengurangan/penambahan HANYA boleh dilakukan di level item (Detail Struk).

**Q: Apa ancaman "Struk Kosong"?**
A: Jika kita memotong Qty item, kita harus memastikan total struk tidak menjadi Rp 0. Jika sebuah struk hanya berisi 1 item dan kita menghapusnya, struk itu akan bernilai Rp 0 (kosong) dan akan menyebabkan *crash* pada pelaporan. Sistem harus mendeteksi dan melewati struk semacam ini (lempar ke *global gap*).

**Q: Jika target pemotongan pajak adalah 20%, haruskah tiap struk dipotong persis 20%?**
A: Secara matematis itu mustahil karena Qty adalah bilangan bulat (integer), tidak bisa pecahan. Potong sebisa mungkin (item paling bawah atau yang harganya mendekati target), sisanya dilempar ke variabel *Global Gap* untuk diakumulasi dan dieksekusi di tempat lain secara acak.

**Q: Mengapa kita harus menggunakan Tabel Tabungan dan Hutang? Mengapa tidak memotong data begitu saja?**
A: Jika data dihilangkan begitu saja, stok barang di gudang akan kacau balau (tidak *balance*). 
- Barang yang dihilangkan dari struk akan disimpan di "Tabungan" (tipe 'tambah').
- Barang fiktif yang disuntikkan ke struk (karena kurang omset pajak) dicatat sebagai "Hutang" (tipe 'kurang'). 
Dalam jangka panjang (tahunan), stok fisik akan selalu seimbang (Keluar - Masuk = 0).

**Q: Apa masalahnya dengan menggunakan "Harga Terbaru" saat menyuntikkan tabungan ke struk?**
A: Jika Sabun ditabung saat harga Rp 1.000, lalu beberapa bulan kemudian harganya naik menjadi Rp 2.000 dan ditarik masuk ke struk, maka secara matematis kita "menciptakan" Omset Rp 1.000 dari udara kosong. Walaupun stok barang seimbang, nilai valuasinya meleset. Klien sudah menyetujui risiko "Harga Mengambang" ini karena fokus utama mereka adalah *balancing* nilai PPN dan kuantitas stok.
